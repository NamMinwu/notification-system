# 03. 규모 추정 및 설계 적합성 검증

> 라이브클래스 알림 시스템의 예상 규모를 추정하고, 그 규모에 현재 설계가 적합한지 검증한다.
> 외부에서 정확한 규모를 알 수 없으므로 **추정 방법론과 그 한계**를 먼저 밝힌다.
> 핵심은 숫자 자체가 아니라 **차수(order of magnitude) → 설계 결정**의 연결이다.

---

## 1. 추정의 한계 (먼저 인정)

비공개 SaaS라 실제 트래픽(DAU, 일 신청 수, 동시 강의 수, 강의당 수강생)은 알 수 없다. 따라서
공개 정보 + 업계 일반론 + 합리적 가정의 조합으로 추정하며, **각 가정에 ±5배 오차**가 있을 수 있다.

## 2. 추정 (Top-down ↔ Bottom-up 교차검증)

- **Top-down**: 한국 중견 SaaS 가정 → DAU 1,000~10,000 → 1인당 1~2건/일 → **일 5,000~15,000건**
- **Bottom-up**: 신청(200~1,000) + 결제(100~700) + D-1 리마인드(1,500~6,000) + 취소(10~100) → **일 2,000~8,000건**
- **교차검증 수렴 → 일 ~5,000건** (중간 시나리오)

### 시나리오

| 시나리오 | 일 발송 | 지속 피크 | 순간 피크 | 필요 동시성* |
|---|---|---|---|---|
| 매우 보수적 | 1,000 | ~0.03/s | ~0.3/s | ~1 |
| **중간 (기준)** | **5,000** | **~0.17/s** | **1~2/s** | **~1~4** |
| 낙관 | 15,000 | ~5/s | ~10/s | ~10 |
| 매우 낙관 | 50,000 | ~15/s | ~30/s | ~30~60 |

\* 필요 동시성 ≈ 지속 피크 × 발송 소요(SMTP 2s 가정). 순간 피크는 Outbox 큐가 흡수.

> 피크 분석: 라이브 강의는 저녁(8~10시)에 몰리지만 D-1 리마인드는 24시간 전 발송이라 자연 분산 →
> 정시 폭주가 약하고 평탄한 부하. 그래서 **지속 피크**(0.17/s)가 설계 기준, 순간 버스트는 큐로 흡수.

## 3. 우리 설계의 처리 용량

- **인스턴스당**: 발송 전용 **고정 풀 `concurrency=16`** → 동시 16건. 실 SMTP 2s 기준 ≈ **8건/s/인스턴스**.
  (Mock=즉시면 폴링 주기·배치에 묶여 ~10건/s.)
- **전역**: `FOR UPDATE SKIP LOCKED`로 인스턴스 간 중복 없이 분배 → **인스턴스 × 8건/s** 선형 확장.
- 평소(중간) 부하에선 풀이 대부분 idle(동시 1~4), 낙관 부하에서 한도(16)까지 차오른다 — **재설계 없이 흡수**.

## 4. 설계 적합성 검증

| 시나리오 | 필요 동시성 | 우리 설계로 |
|---|---|---|
| 매우 보수적 | ~1 | 1인스턴스 ✓ |
| **중간 (기준)** | ~1~4 | **인스턴스 2~3대(HA 겸) + 고정 풀 16** ✓ (풀 일부만 사용, 여유) |
| 낙관 | ~10 | 인스턴스 1~2대로 풀이 차올라 흡수 ✓ |
| 매우 낙관 | ~30~60 | 임계 초과 → **발송 tx-밖 + sender 멱등 + PgBouncer** 도입 |

### 구성 요소별 결론

| 요소 | 중간 | 10배(낙관) | 근거 |
|---|---|---|---|
| **DB Outbox + Polling** | ⭐ 적합 | ⭐ 적합 | 15/s도 DB 폴링 병목(수백/s)과 차원이 다름 → **Kafka는 과대** |
| **단일 PostgreSQL** | ⭐ 적합 | ⭐ 적합 | 일 INSERT 5k + 폴링 17k/일 + UPDATE 15k/일 → CPU 여유. 샤딩·복제 불필요 |
| **고정 풀(16)** | ⭐ 적합 | ⭐ 적합 | 중간 기준 + 낙관(~10) 여유. Virtual Thread는 "의도적 제한" 목표와 반대라 불필요(오버스펙) |
| **5초 Polling** | ⭐ 적합 | ⭐ 적합 | SLA 1분 >> 5초. 5초당 평균 0.3건/피크 5~10건 → 배치 50으로 충분 |
| **Lease 10분 + Sweeper 1분** | ⭐ 적합 | ⭐ 적합 | 불변식 `lease ≥ ⌈batch/conc⌉×sender-timeout` 충족(⌈50/16⌉×2분=8분) |
| **다중 인스턴스(2~3)** | ⭐ 적합 | ⭐ 적합(필요 시 ↑) | SKIP LOCKED 수평 확장. `인스턴스 × 풀 ≤ PG max_connections(100)` |

## 5. 확장 노브 (부하 증가 시 순서)

1. **인스턴스 추가** (SKIP LOCKED 수평) — 낙관 전까지 1차 수단.
2. **폴링 주기↓ / 배치↑** — 백로그 지연 단축. (매우낙관 75/5s면 배치 상향)
3. **`worker.concurrency` 상향** — 인스턴스당 동시성 확대. 단 `lease`/`hikari pool` 동반 재산정.
4. **발송 tx-밖 + sender 멱등 + PgBouncer** — 동시성 수십+ (매우낙관).
5. (그 이상, 수백/s↑) **CDC → Kafka** — 본 추정 범위 밖. Dispatcher/Sender 추상화로 전환 경로 확보.

## 6. 신뢰도 & 결론

- 신뢰도 "낮음" 가정이 많아 **±5배 오차** 가능. 정확한 숫자보다 **차수**가 중요.
- **결론**: 현실 규모(~5k/일)는 **DB Outbox + 고정 풀(16) + 2~3 인스턴스**로 정확히 커버되고,
  **10배 성장(낙관)까지 재설계 없이** 대응한다. Kafka·Virtual Thread·샤딩은 현 규모에 과대이며,
  **언제·어떻게 도입할지**(위 노브 순서)를 명시해 전환 경로를 열어뒀다.
