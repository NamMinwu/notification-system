# 02. 요구사항 해석 및 개선 의견

> 과제 추가 제출물 ②. 모호한 요구사항을 어떻게 해석했고, 어떤 가정을 두었으며,
> 현재 설계의 한계와 개선하고 싶은 점을 정리한다.

---

## A. 요구사항 해석

명세에 명시되지 않아 **판단이 필요했던 지점**들과, 내가 내린 결정·가정을 적는다.

### A-1. "비즈니스 영향 X, 단 단순 무시 금지"

- **해석**: 외부 발송(SMTP) 실패가 비즈니스 트랜잭션(결제·수강신청)을 롤백시키면 안 된다.
  동시에 try-catch로 예외를 삼키는 것은 금지 → 실패는 **상태와 로그로 추적 가능**해야 한다.
- **결정**: Transactional Outbox. 알림은 비즈니스와 같은 트랜잭션에 `PENDING`으로 INSERT만 하고,
  외부 발송은 Worker가 트랜잭션 밖에서 수행. 실패는 `FAILED`/`DEAD_LETTER` + `last_error_*`로 기록.
- **인정한 한계**: 알림 INSERT 자체가 실패하면 비즈니스도 롤백된다(같은 DB). 그러나 이는
  DB 장애 상황이라 비즈니스도 어차피 실패하는 "운명 공동체"이므로 수용했다.

### A-2. "알림 상태를 적절히 정의" — 상태 세분도

- **해석**: 상태 수에 정답은 없고, 운영 가시성과 단순성의 균형 문제.
- **결정**: 기본 5-state(PENDING/PROCESSING/SENT/FAILED/DEAD_LETTER)에,
  선택 구현(예약 발송)의 **취소**를 위해 `CANCELLED`를 더한 **6-state**.
  - `PROCESSING`을 분리한 이유: 좀비 감지 + 다중 Worker 락 표현.
  - `FAILED`와 `DEAD_LETTER`를 분리한 이유: "자동 재시도 대기"와 "운영자 개입 필요"는 다른 도메인.

### A-3. "재시도 가능"

- **해석**: 단순히 "재시도한다"가 아니라 *언제·몇 번·어떤 간격으로*, 그리고 *모든 실패가 재시도 대상인가*를 정해야 한다.
- **결정**: Exponential backoff + jitter, 5회. **실패를 retryable / permanent로 분류**하여
  영구 실패(잘못된 이메일 등)는 재시도 없이 즉시 DEAD_LETTER.
- **가정**: 일시 장애는 보통 ~30분 내 복구된다(1+2+4+8+16 ≈ 31분). 수치는 운영 데이터로 튜닝.

### A-4. "중복 발송 방지 + 동시 요청 고려"

- **해석**: 두 가지 다른 중복을 모두 막아야 한다.
  - 논리적 중복: 같은 이벤트가 두 번 등록 (webhook 재전송 등)
  - 물리적 동시성: 같은 요청이 동시에 도착 (더블 클릭, 다중 인스턴스)
- **"동일 이벤트"의 정의** (명세에 없어 직접 정함):
  `event_id + recipient_id + channel + notification_type` 4요소 조합.
  - 같은 결제·같은 사용자·EMAIL 2회 → 중복
  - 같은 결제·같은 사용자·EMAIL + IN_APP → 다른 알림 (채널이 다름)
  - 같은 결제·다른 사용자 → 다른 알림
- **결정**: 위 4컬럼에 **DB UNIQUE 제약** → 동시성을 애플리케이션이 아니라 **DB에 위임**.
  등록 단계 중복은 UNIQUE로, 발송 단계 중복은 `SKIP LOCKED`로 분리해 처리.
- **가정**: `event_id`가 없는 알림(관리자 단발 발송)은 중복 검사 대상이 아니다
  (NULL은 distinct로 취급되어 자연히 dedup에서 제외).

### A-5. "비동기 / 메시지 브로커 없이 / 운영 전환 가능"

- **해석**: 인메모리가 아니라 **영속 큐**여야 하고(재시작 유실 X), 나중에 Kafka로 갈 수 있어야 한다.
- **결정**: DB Outbox + Polling Worker. `NotificationDispatcher` 인터페이스로 추상화하여
  향후 Consumer 구현체로 교체 가능. DB 테이블은 그대로 두고 CDC로 Kafka 전환하는 진화 경로.

### A-6. "운영 시나리오 (좀비 복구 / 재시작 / 다중 인스턴스)"

- **해석**: 세 가지는 모두 "장애 상황에서도 정확히 한 번 처리"라는 한 주제의 다른 측면.
- **결정**: Lease + Sweeper(좀비), DB 영속성(재시작), `FOR UPDATE SKIP LOCKED`(다중 인스턴스).

### A-7. 인증/인가

- **해석**: 명세가 "간략히 처리 무방, userId 헤더 허용" → 핵심 로직에 집중.
- **결정**: `X-User-Id` 헤더로 사용자 식별. 단, **본인 알림만 조회/읽음 처리** 권한 검증은 유지
  (다른 사람 알림을 읽음 처리하는 것은 막는다). 수동 재시도는 운영자(ADMIN) 권한 가정.

### A-8. 선택 구현 4종

| 항목 | 해석 / 결정 |
|---|---|
| 발송 스케줄링 | `scheduled_at` 컬럼 1개로 즉시·예약 발송을 같은 코드 경로로. 과거 시각은 즉시 발송 허용 |
| 템플릿 관리 | DB 저장 + Mustache 렌더링 → 운영 중 문구 수정·A/B·다국어 확장 |
| 읽음 처리 | 멱등 UPDATE(`WHERE is_read = false`) → 다중 기기 동시 요청에도 `read_at`은 첫 시각으로 고정 |
| 수동 재시도 | `retry_count` 유지 → 무한 루프 자동 차단 |

## B. 개선 의견 (자유 기술)

현재 설계는 과제 범위에 맞춰 의도적으로 단순화한 부분이 있다. 실제 운영 시 개선하고 싶은 점들이다.

### B-1. 명세 자체에 대한 제안

1. **"동일 이벤트"의 정의를 명세가 못 박아야 한다.**
   채널/타입을 키에 포함할지에 따라 중복 판정이 완전히 달라진다. 본 구현은 4요소로 가정했지만,
   비즈니스가 "한 이벤트당 사용자에게 모든 채널 합쳐 1건"을 원한다면 키에서 channel을 빼야 한다.

2. **알림 수신 동의/거부(opt-out) 정책이 명세에 없다.**
   실제로는 사용자가 마케팅 알림을 거부했는지, quiet hours(야간 미발송)인지 등을
   발송 직전 확인해야 한다. 현재는 무조건 발송한다.

### B-2. 현재 설계의 한계와 보완책

3. **Sender 레벨 idempotency 부재.**
   Lease 만료 edge case(원래 Worker의 발송이 뒤늦게 성공)에서 중복 발송이 가능하다.
   운영에서는 외부 발송에 멱등 키(message-id)를 실어 "최소 1회"를 "정확히 1회"에 가깝게 만들어야 한다.

4. **Polling의 본질적 지연·부하.**
   5초 주기라 등록~발송에 최대 5초 지연이 있고, 빈 polling도 DB를 친다.
   트래픽이 커지면 빈 polling backoff(연속 빈 결과 시 주기 확대)나 CDC+Kafka 전환을 도입하고 싶다.

5. **우선순위 없음(FIFO).**
   결제 확정 같은 중요 알림이 마케팅 알림 뒤에 밀릴 수 있다. `priority` 컬럼 + 정렬,
   또는 채널/타입별 별도 워커 풀로 분리하는 개선을 고려할 수 있다.

6. **수신자별 rate limiting 없음.**
   버그나 잘못된 배치로 한 사용자에게 알림이 폭발할 수 있다. 수신자당 발송 상한(예: 시간당 N건)을
   두는 안전장치가 필요하다.

### B-3. 운영 가시성 / 확장

7. **DEAD_LETTER 모니터링·알림.**
   에러 코드별 분포, 시간당 발생량, 오래 방치된 건을 대시보드/알람으로 노출해야 운영자가 즉시 대응한다.
   본 과제는 조회 쿼리 수준까지만 설계하고 실제 대시보드는 범위 밖으로 두었다.

8. **읽음 상태 실시간 동기화.**
   PC에서 읽으면 모바일의 미읽음 카운트도 즉시 줄어드는 UX는 SSE/WebSocket이 필요하다.
   본 구현은 "읽음이 정확히 한 번 기록된다"까지만 보장한다(실시간 푸시는 범위 밖).

9. **payload(JSONB) 스키마 검증 강화.**
   유연성을 얻은 대신 타입 안전성을 잃었다. 타입별 JSON Schema를 두어 등록 시점에 검증하면
   "템플릿이 요구하는 변수 누락"을 발송 전에 더 일찍 잡을 수 있다.

10. **클라이언트 `Idempotency-Key` 헤더 옵션.**
    `event_id`가 없는 단발 요청도 클라이언트 재시도 안전성을 갖도록, 헤더 기반 멱등 키를
    보조 수단으로 받는 것을 고려할 수 있다(현재는 4컬럼 제약만 사용).

## C. 요약

- 모호한 지점은 **가정을 명시**하고 가장 단순하면서 요구사항을 만족하는 선택을 했다.
- 과제 범위를 벗어나는 것(실시간 동기화, 대시보드, opt-out 등)은 **의도적으로 제외**하되,
  어디까지가 경계인지와 개선 방향을 함께 적었다.
